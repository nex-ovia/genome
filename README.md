# nexovia · `genome`

[![CI](https://github.com/nex-ovia/genome/actions/workflows/ci.yml/badge.svg)](https://github.com/nex-ovia/genome/actions/workflows/ci.yml)
[![Release](https://img.shields.io/github/v/release/nex-ovia/genome?include_prereleases&label=release)](https://github.com/nex-ovia/genome/releases)
[![License: BSL 1.1](https://img.shields.io/badge/license-BSL%201.1-blue.svg)](LICENSE)

**Understand any system without reading its code.** `genome` turns one editable `nexovia.toml`
into a self-contained, offline HTML map of your project — its architecture, every module, what each
part does and *why* — so architects, engineers, and executives can **see it, question it, and
approve it** in the browser.

> ⚠️ **Experimental** — early MVP, released for validation and feedback. The schema, CLI, and HTML
> may change without notice. Prebuilt for **Linux, macOS (Intel & Apple Silicon), and Windows** on
> the [Releases](https://github.com/nex-ovia/genome/releases) page.

👉 **See it before installing anything:** [open this repo's own genome, rendered »](https://htmlpreview.github.io/?https://github.com/nex-ovia/genome/blob/main/mock/report.html)
(that HTML file is generated by running `genome` on this very repo.)

## Install

**Download a prebuilt binary** (recommended) — a single self-contained file, no runtime, no
dependencies, fully offline. Grab the archive for your OS from the
[latest release](https://github.com/nex-ovia/genome/releases) and extract it:

| OS | Asset |
|----|-------|
| Linux x86_64 | `…-x86_64-unknown-linux-musl.tar.gz` |
| macOS (Apple Silicon) | `…-aarch64-apple-darwin.tar.gz` |
| macOS (Intel) | `…-x86_64-apple-darwin.tar.gz` |
| Windows x86_64 | `…-x86_64-pc-windows-msvc.zip` |

```sh
# example: macOS (Apple Silicon)
tar xzf genome-*-aarch64-apple-darwin.tar.gz
sudo mv genome-*/genome /usr/local/bin/     # put it on your PATH
genome render nexovia.toml > report.html    # see "Try it" below
```

*Or build from source* (see [Development](#development)).

## Try it in 60 seconds

This repo describes **itself** as a genome, so you can render it the moment you have the binary:

```sh
git clone https://github.com/nex-ovia/genome && cd genome
genome render nexovia.toml > report.html
open report.html        # macOS · use `xdg-open` on Linux, `start` on Windows
```

You get an offline HTML map of the project. Switch the **Exec / Arch / Eng** lenses; expand any
module to see its summary, rationale (`why`), status, open questions, and approvals.

Prefer a plain-English, non-code example? A product one-pager renders the same way:

```sh
genome render examples/onepager.nexovia.toml > onepager.html
```

Now drive the loop — **understand → ask → approve** — all written back into the TOML:

```sh
genome validate nexovia.toml                                       # schema + honesty gates
genome ask render_html "why is this on the surface layer?" --role arch
genome approve "*" --role arch --decision approved --note "scope looks right"
genome render nexovia.toml > report.html                           # the Q&A + approval now show
```

Optional **offline AI enrichment** — fills blank `summary`/`why` fields with an embedded model,
fully offline (the model is fetched once, then never touches the network):

```sh
cargo build --release --features enrich
genome enrich examples/onepager.nexovia.toml
```

## What you can use it for

- **Review a design before any code exists.** Turn a spec or one-pager into a structured, reviewable
  map (`genome from-doc spec.md nexovia.toml`), then question and approve it.
- **Onboard onto an unfamiliar system.** Read the plain-language `summary` / `why` per module instead
  of spelunking the source.
- **Sign-off & governance.** Executives and architects approve per-module or the whole snapshot;
  every question and decision is captured in version control, not a chat thread.
- **Keep understanding from drifting.** Re-render after changes — the HTML is always a pure
  projection of the source of truth, never a stale doc.

## How it works

The TOML files are the **database / control plane**; the HTML is a **static, read-only window** onto
them. Nothing runs in the page; re-render to refresh.

```
   document ──▶ genome from-doc ──▶ nexovia.toml  (design-first nodes)
                                          │
                                          ▼  genome enrich   (offline, embedded model — optional)
                              plain-language summary / why per node   (by=llm + confidence)
                                          │
   genome ask <node|*> "question"  ──┐    │    ┌──  genome approve <node|*> --decision …
   (writes questions facet)          ▼    ▼    ▼    (writes approvals facet)
                              ┌────────────────────────┐
                              │      genome render       │ ──▶ report.html  (static, offline)
                              └────────────────────────┘
```

**Why you can trust it**

- **Honest by construction.** Every semantic field carries provenance (`by` = human / scan / llm) and
  a confidence. Machine guesses never overwrite a human edit — provenance is sticky.
- **Offline & network-free by default.** The default binary has no HTTP client and no inference engine
  in its dependency graph (CI enforces this). Comprehension stays local.
- **The genome is the plan.** This project is built with itself: its own roadmap, components, and
  policies live in the genome and render to the same HTML you see above.

## Status

An early MVP that proves the loop — **understand → ask → approve** — fully offline:

- ✅ `genome render` → self-contained HTML (Exec / Arch / Eng lenses)
- ✅ `genome validate` → schema + honesty + integrity gates
- ✅ `genome from-doc` → document to design-first nodes
- ✅ `genome enrich` → offline embedded-model enrichment (`enrich` feature)
- ✅ `genome ask` / `genome approve` → questions + approvals as TOML facets

**Designed for, built later:** codebase ingest (tree-sitter), LSP surface, skeleton scaffolding,
freeze + drift tracking, an MCP server for agents, and optional cloud enrichment.

## Development

Build from source (needs a Rust toolchain):

```sh
cargo build --release                 # default: offline, no inference engine
cargo build --release --features enrich   # adds the embedded GGUF model (needs cmake + clang)
cargo test                            # includes the render-fidelity check
```

Repository layout:

```
nexovia.toml            the anchor — project identity + architecture spine + [include]
project/                this project's generated-then-refined fragments
  components.toml         modules + links (the structural reality)
  policies.toml           local policies (override/extend the org bases)
  lifecycle.toml          the phased roadmap
  questions.toml          questions raised against nodes or the whole snapshot
  approvals.toml          approval / rejection decisions, per node + per snapshot
org/                    SHARED, centrally-owned standards — inherited, never copied
schema/nexovia-2.toml   the formal grammar `validate` enforces (schema = "nexovia/2")
src/                    the Rust workspace (genome contract, ingest, enrich, render, validate, cli)
examples/               worked genomes (incl. a non-code document genome)
mock/report.html        this repo's own genome, rendered — regenerated by `genome render`
```

Humans rarely edit the TOML directly — only to set a guardrail, which is sticky (`by=human`) and
never overwritten by an agent or a re-scan.
